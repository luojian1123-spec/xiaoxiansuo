# 线索管理系统项目规划方案

## 1. 需求结论与可行性判断

补充维护说明：

- 当前原型资产以 `prototype/` 目录下的最终 HTML / CSS / JS 文件为唯一事实来源。
- 历史页面生成脚本 `prototype/generate_pages.py` 已停用，后续不再承担页面生成或同步职责。

基于《中小型企业线索管理系统产品规划（产品经理视角）》的内容，这个项目本质上不是一个简单的“线索录入工具”，而是一套围绕 **线索收集 -> 进入公海 -> 异常处理 -> 分发 -> 跟进 -> 转化 -> 复盘** 的轻量化业务系统。

核心价值有 4 个：

1. 统一多来源线索，解决线索分散和重复录入问题。
2. 建立标准化流转机制，解决异常处理、分发、跟进无规则的问题。
3. 提供基础权限和数据报表，解决管理层无法追踪和复盘的问题。
4. 支撑 PC + 移动端协同，满足管理员和一线员工不同场景下的使用需求。

结论：**项目可做，且有明确 MVP 路径，但不建议一开始全量铺开。**

原因：

1. 文档中的业务链路完整，但模块较多，尤其是“广告 API 对接、转介绍、消息通知、多端同步、自定义字段、报表、权限、复杂审批配置”会显著抬高首期复杂度。
2. 真正的难点不在页面数量，而在 **状态流转、权限边界、分发规则、消息时效、数据一致性**。
3. 如果团队人数有限，建议采用 **核心链路先落地，增值能力后迭代** 的方式推进，否则首期周期和风险都会失控。

## 2. 推荐的 MVP 边界

### 2.1 建议首期一定要做

围绕主流程优先保证“能收、能进公海、能处理异常、能分、能跟、能看”。

- 线索收集
  - 员工手工录入
  - Excel 批量导入
  - URL/二维码表单收集
- 线索公海
  - 线索列表
  - 基础筛选
  - 系统异常标记
  - 合并 / 继续流转 / 无效归档
- 线索分发
  - 手动分发
  - 轮循自动分发
  - 公海抢单
  - 分发对象类型为主管时，由主管二次分配
  - 员工退回公海
- 线索跟进
  - 跟进记录
  - 线索状态流转
  - 跟进提醒基础能力
  - 消息中心 + 统一消息配置页
  - 公众号模板消息基础接入能力（可选增强通道）
- 权限与组织
  - 管理员 / 经理 / 员工三类基础角色
  - 部门归属
  - 数据查看范围控制
- 报表
  - 来源统计
  - 员工跟进统计
  - 转化漏斗基础图
- 终端
  - PC 管理端
  - 移动员工端（二选一：H5 优先，或小程序优先）

### 2.2 建议放到 P1/P2

- 加权分发 / 条件分发
- 用户转介绍
- 复杂审批配置
- 高级自定义报表
- 数据预警
- 会员系统 / CRM 对接
- 积分、佣金、奖励系统
- 广告商 API 对接（建议最后实施）

### 2.3 可行性建议

如果你的团队规模不大，**首期不要同时做 H5 和小程序双移动端**。建议如下：

- 如果业务重微信生态：优先小程序，H5 后置。
- 如果追求开发效率和跨端复用：优先 H5，后续再封装小程序。

## 3. 业务拆解与系统边界

### 3.1 核心业务对象

- 线索 Lead
- 线索来源 Source
- 线索分类 Tag / Category
- 异常处理记录 ExceptionRecord
- 分发记录 AssignRecord
- 跟进记录 FollowRecord
- 组织 Department / Team
- 用户 User / Role / Permission
- 通知 Notice
- 报表统计 ReportSnapshot

### 3.2 核心状态流转

建议尽早定义统一状态机，避免后续逻辑混乱。

- 首期建议将状态分为 `系统流转状态` 与 `业务结果状态` 两类，但同一时刻仅保留一个主状态，避免页面和接口出现双口径。
- 推荐主状态链路：
  - 新建
  - 异常待处理
  - 待归组
  - 已归组待分发
  - 待主管分配
  - 待抢单
  - 已分发
  - 跟进中
  - 已预约
  - 已转化
  - 已放弃
  - 无效线索
  - 退回待处理
  - 退回待分发
- 关键控制点：
  - `异常待处理` 不能直接跳到 `已分发` 或 `跟进中`
  - `待归组` 前不能发起员工分发
  - `已分发` 与 `跟进中` 要区分，避免“刚分配即算已跟进”
  - `已转化`、`无效线索` 视为默认终态
  - `已放弃` 属于业务终态，但允许管理员重新激活

### 3.3 首期最关键的规则

- 去重规则：手机号优先，姓名 + 手机号辅助。
- 异常处理规则：哪些异常自动标记、谁可以处理、处理后去哪里。
- 分发规则：手动、轮循、抢单、退回、公海重分配。
- 在存在录入专员或客服集中录入场景时，支持“先分配给主管，再由主管分给员工”的二段式承接链路。
- 规则组需同时具备“入组规则 + 组内分发策略”两段配置，避免只有分发策略、没有归组依据。
- 规则组配置需配套模板、测试和发布前强校验，降低中小企业管理员误配成本。
- 未命中规则组的线索应进入独立未归组队列，由系统给出推荐规则组并支持人工批量归组。
- 数据权限：员工只能看自己的线索；经理看本组；管理员看全量。
- 操作留痕：异常处理、分发、退回、状态变更、跟进都必须记录日志。

## 4. UI 规划

### 4.1 UI 设计目标

这个系统的 UI 不追求“炫”，而追求：

- 高信息密度但不压迫
- 高频操作路径短
- 列表、筛选、详情、跟进切换顺滑
- 移动端录入快、提醒清楚、状态明显

### 4.2 PC 端主要页面

- 登录 / 首页驾驶舱
- 线索公海列表页
- 线索详情页
- 异常处理页
- 分发配置页
- 员工管理 / 角色权限页
- 表单配置页
- 导入中心
- 报表中心
- 消息通知中心
- 消息配置中心

### 4.3 移动端主要页面

- 工作台
- 我的线索
- 待跟进列表
- 抢单池
- 新增线索
- 线索详情
- 快速记录跟进
- 退回线索
- 提醒中心
- 我的
- 用户提交表单

补充说明：

- 员工端小程序底部导航建议固定为 4 个一级入口：`工作台 / 线索 / 录入 / 我的`
- 提醒中心作为二级页面放入工作台，不单独占用底部 Tab
- 抢单池作为条件型二级页面，仅在后台开启抢单模式后对符合范围的员工显示入口

### 4.4 UI 工作拆分建议

- 信息架构
  - 导航结构
  - 页面层级
  - 列表与详情的关系
- 交互原型
- 异常处理流
  - 分发流
  - 跟进流
  - 退回流
- 视觉设计
  - PC 管理后台设计规范
  - 移动端组件规范
  - 状态色、标签、表格、表单、弹窗、消息规范
- 设计产出物
  - 全局设计规范
  - 高保真页面
  - 交互说明
  - 异常态与空状态说明

### 4.5 UI 优先级

先做这些页面，能直接支撑研发开工：

1. 登录 / 首页
2. 线索列表
3. 线索详情抽屉或详情页
4. 异常处理弹窗 / 页
5. 分发弹窗 / 页
6. 跟进记录页
7. 员工端我的线索 + 跟进录入

## 5. 前端规划

### 5.1 前端架构建议

建议拆为两个前端应用：

- PC 管理端
- 移动端（H5 或小程序）

建议能力层统一：

- 统一 API SDK
- 统一权限判断
- 统一状态字典
- 统一消息中心协议

### 5.2 前端模块拆分

#### PC 管理端

- 认证登录与权限路由
- 线索列表 / 筛选 / 批量操作
- 线索详情 / 操作日志 / 跟进记录
- 异常处理模块
- 分发模块
- 导入模块
- 组织架构 / 角色权限模块
- 报表模块
- 系统设置模块

#### 移动端

- 工作台
- 我的线索
- 待办提醒
- 抢单池
- 新增线索录入
- 跟进记录录入
- 状态变更
- 退回公海
- 个人数据与绑定状态
- 用户线索提交表单

### 5.3 前端开发重点

- 列表性能和筛选体验
- 大量状态标签与权限按钮控制
- 表单动态字段渲染
- 跟进记录快速录入体验
- 多端消息提醒一致性
- 异常处理和弱网兼容

### 5.4 前端可并行拆分

- 前端 A：PC 管理端基础框架、登录、权限、公共组件
- 前端 B：线索业务模块（列表、详情、异常处理、分发、跟进）
- 前端 C：移动端（H5/小程序）与用户提交表单

如果只有 2 名前端：

- 前端 1：PC 端
- 前端 2：移动端 + 公共接口联调

## 6. 后端规划

### 6.1 后端职责核心

后端是这个项目的主复杂度来源，重点不只是 CRUD，而是规则引擎和数据一致性。

### 6.2 后端模块拆分

- 用户、组织、角色、权限
- 线索收集
  - 手工录入
  - 批量导入
  - 外部表单提交
- 去重与校验
- 公海管理
- 异常处理流程
- 分发引擎
  - 手动分发
  - 轮循分发
  - 退回重分发
- 跟进记录
- 状态流转引擎
- 通知中心
- 报表统计
- 操作日志 / 审计日志
- 文件上传
- 系统配置

### 6.3 数据库设计重点

核心表建议至少包含：

- users
- roles
- permissions
- departments
- leads
- lead_custom_fields
- lead_import_tasks
- lead_exceptions
- lead_assignments
- lead_follow_records
- lead_status_logs
- lead_invalid_reasons
- notifications
- form_templates
- report_aggregates

### 6.4 后端技术重点

- 状态机设计
- 权限模型设计
- 审计日志完整性
- 导入任务异步化
- 报表统计口径统一
- 消息通知解耦
- 对外 API 扩展能力预留

### 6.5 公众号模板消息实现依赖

- 员工与微信接收身份绑定能力，至少支持员工账号与公众号接收身份一对一绑定
- 管理端配置能力，用于维护公众号 AppID、模板映射、提醒开关和兜底策略
- 模板配置能力，支持按提醒类型维护模板编号、字段映射和启停状态
- 发送日志能力，记录发送时间、目标员工、线索、模板类型、结果和失败原因
- 失败兜底能力，公众号发送失败时自动降级为站内消息与移动端提醒
- 重复触达控制，避免同一线索同类提醒在短时间内重复发送

补充说明：

- 首期版本的消息能力应按 `站内消息 + 小程序提醒 + 可选公众号模板消息` 的统一架构设计，而不是按单一公众号能力建设
- 公众号模板消息仅作为可选增强通道，默认适用于线索分配提醒、异常处理结果提醒、线索退回结果提醒
- 跟进待办提醒、主管超时未分配提醒等高频催办类提醒，首期不建议默认启用公众号模板消息

### 6.6 后端可并行拆分

- 后端 A：用户权限、组织、基础配置、认证
- 后端 B：线索主链路（收集、公海、异常处理、分发、跟进）
- 后端 C：导入任务、通知、报表统计、日志

如果只有 2 名后端：

- 后端 1：权限 + 线索主链路
- 后端 2：导入、报表、消息、运维支撑

## 7. 测试规划

### 7.1 测试重点

这个系统测试重点不是单页面功能，而是“流程正确”和“权限正确”。

### 7.2 测试范围拆分

- 功能测试
  - 收集、公海、异常处理、分发、跟进、退回、转化、报表
- 权限测试
  - 不同角色的数据范围
  - 按钮是否可见
  - 越权接口拦截
- 流程测试
  - 线索完整生命周期
  - 重复线索处理
  - 分发后退回再分发
- 接口测试
  - 导入接口
  - 外部表单接口
  - 消息通知接口
  - 公众号模板消息接口
- 兼容性测试
  - PC 浏览器
  - H5 / 小程序基础机型
- 性能测试
  - 大列表查询
  - 批量导入
  - 报表统计
- 安全测试
  - 登录与鉴权
  - 敏感信息保护
  - 导出权限控制

### 7.3 核心测试用例必须覆盖

- 重复手机号导入
- 异常处理后是否进入正确状态
- 轮循分发是否均衡
- 员工是否只能看到本权限范围内线索
- 线索退回后是否可重新分发
- 跟进记录是否可追溯且不可被普通员工篡改
- 报表口径是否与明细一致
- 导出数据是否脱敏或受权限控制

### 7.4 测试节奏建议

- 需求评审时介入测试
- UI 出稿后同步测试场景
- 接口定义完成后先做接口用例
- 每个迭代末做业务回归
- 上线前做一次全链路冒烟 + 权限专项

## 8. 项目阶段拆分

## 8.1 阶段 0：需求定稿与方案设计（1-2 周）

目标：把“能做什么”和“先做什么”彻底定清楚。

输出：

- PRD 定稿
- 业务流程图
- 状态流转图
- 权限矩阵
- 字段清单
- 原型图
- 技术方案
- 迭代计划

参与角色：

- 产品经理
- UI/UX
- 前端负责人
- 后端负责人
- 测试负责人

### 8.2 阶段 1：MVP 开发（4-6 周）

目标：打通最小闭环。

范围：

- 线索收集
- 公海与异常处理
- 手动 / 轮循分发
- 跟进记录
- 基础权限
- PC 端
- 一个移动端
- 基础报表

里程碑：

1. 第 1 周：框架、登录、权限、基础数据模型
2. 第 2 周：收集、公海、异常处理
3. 第 3 周：分发、退回、跟进、公众号模板消息
4. 第 4 周：移动端、报表、联调
5. 第 5-6 周：测试、修复、试运行

### 8.3 阶段 2：优化迭代（3-5 周）

目标：提高效率和可用性。

范围：

- 抢单模式
- 跟进提醒
- 自定义字段
- 基础消息通知
- H5 / 小程序补齐
- 报表优化

### 8.4 阶段 3：增值能力（4-8 周）

目标：增强竞争力和扩展性。

范围：

- 转介绍
- 高级报表
- 数据预警
- 复杂审批配置
- 外部 CRM / 会员系统对接

### 8.5 阶段 4：外部广告平台对接（单独收尾阶段，2-4 周）

目标：在主业务链路稳定后，再接入外部广告线索来源，降低首期不确定性。

范围：

- 广告 API 对接
- 同步异常告警
- 对接配置后台
- 数据映射与字段校验
- 失败补偿与手工兜底导入

## 9. 人员安排建议

### 9.1 标准配置

建议配置：

- 产品经理 1
- UI/UX 1
- 前端 2
- 后端 2
- 测试 1
- 项目负责人或技术负责人 1（可兼任）

这是比较稳妥的 7 人配置。

### 9.2 精简配置

如果预算有限，可采用：

- 产品经理 1
- UI/UX 1
- 前端 2
- 后端 2
- 测试 1（可后半程介入）

这是 6 人配置，MVP 可做，但要严格控范围。

### 9.3 高风险配置

如果低于 5 人同时还要求 PC + H5 + 小程序 + 报表 + API 对接一起上，风险很高：

- 需求变更会直接拖慢进度
- 联调时间会明显增加
- 测试覆盖不足
- 首期上线质量不稳定

## 10. 推荐开发顺序

建议按依赖关系推进：

1. 权限、用户、组织、基础字典
2. 线索主表、状态流转、日志
3. 收集入口与导入
4. 公海与异常处理
5. 分发逻辑
6. 跟进记录
7. 移动端录入与提醒
8. 报表统计
9. 高级规则和外部对接

## 11. 关键风险与应对

### 11.1 最大风险

- 需求过宽，首期范围失控
- 状态和权限定义不清，导致返工
- 多端同时推进，联调成本过高
- 报表口径后补，导致数据不一致
- API 对接过早介入，拖累主链路
- 外部广告平台接口变化频繁，容易影响主版本节奏

### 11.2 应对建议

- 首期只保核心闭环
- 在开发前先定状态机和权限矩阵
- 先统一接口协议和字段字典
- 报表从第一天就定义统计口径
- 第三方广告 API 单独放到最后阶段

## 12. 结论

这份需求适合做成一个 **分阶段落地的中小企业线索管理产品**，首期重点不是“功能多”，而是把主链路做稳。

最可行的方案是：

- 首期先做 PC 管理端 + 一个移动端
- 先打通收集、公海、分发、跟进、基础报表
- 把广告 API、转介绍、复杂分发、高级报表放到后续
- 用 6-7 人团队推进，MVP 周期控制在 6-8 周更稳妥

如果项目要顺利落地，最先需要敲定的不是页面，而是：

1. 线索状态流转图
2. 角色权限矩阵
3. MVP 范围清单
4. 多端优先级顺序
